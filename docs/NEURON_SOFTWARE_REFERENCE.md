# Neuron Software Reference Workflow

## Purpose

The Capsule Command Console exposes a **Software Reference** screen that shows what a Neuron currently has installed versus what the selected reference expects. Field teams should not need to log into that command console just to answer basic survey questions.

SysAdminSuite now carries an offline baseline and a comparison tool so a tech can answer:

- Which software reference is this Neuron expected to match?
- Which firmware packages are expected?
- Which DDI packages are expected?
- Is an observed package snapshot missing anything?
- Does a package version differ from the reference?
- Are there extra packages that distinguish this device from another?

## Tool

```text
GetInfo/Get-NeuronSoftwareReference.ps1
```

### Emit the expected package set

```powershell
powershell.exe -File .\GetInfo\Get-NeuronSoftwareReference.ps1
```

This generates:

- expected CSV
- comparison CSV with `ReferenceOnly` status
- summary JSON
- HTML report when the suite HTML helper is available

### Compare an observed package snapshot

```powershell
powershell.exe -File .\GetInfo\Get-NeuronSoftwareReference.ps1 `
  -ObservedPath .\GetInfo\Config\NeuronObservedPackages.example.csv `
  -NeuronHost A0F5097FE1C4
```

Observed CSV schema:

```csv
Category,Name,Version
firmware,Application binaries packages,11.8.0.328
ddi,AspectA,5.1.3.10
```

Comparison statuses:

| Status | Meaning |
|---|---|
| `OK` | Observed package exists and version matches the reference. |
| `Missing` | Expected package was not found in the observed snapshot. |
| `VersionMismatch` | Package exists, but the version differs. |
| `Extra` | Observed package is not part of the selected reference. |
| `ReferenceOnly` | No observed snapshot was supplied, so the output is a clean baseline. |

## Remote emulation target

The first safe implementation is **adapter-based**, not console-login-based.

The comparison engine accepts observed package data from any future collector as long as it outputs this shape:

```text
Category,Name,Version
```

Future collectors can be added without rewriting the comparison logic:

1. **Manual observation adapter**
   - Field tech records package rows from the console view or export.
   - Lowest risk.

2. **Command-console export adapter**
   - Pulls data from an approved export/API endpoint if one exists.
   - Must avoid credential storage in scripts.

3. **Neuron network inventory adapter**
   - Joins host/MAC/serial identity from `Get-NeuronNetworkInventory.ps1` with the software reference result.
   - Useful for distinguishing devices during surveys.

4. **QRTasks field adapter**
   - A short QR launcher could run a local collector if the Neuron or connected workstation exposes package data locally.
   - The QR should point to a script, never embed the script.

## Survey use case

For each Neuron row in the deployment tracker, the eventual survey output should include:

| Field | Purpose |
|---|---|
| Neuron Hostname | Human-readable device identity. |
| Neuron MAC | Best hardware identity when hostname is uncertain. |
| Neuron Serial | Asset confirmation. |
| Software Reference | Expected baseline such as `11.8.0.328`. |
| Firmware Match | `OK`, `Missing`, or `VersionMismatch`. |
| DDI Match | `OK`, `Missing`, or `VersionMismatch`. |
| Missing Packages | Fast triage column. |
| Version Mismatches | Fast triage column. |
| Extra Packages | Helps distinguish one device from another. |
| Evidence Path | Link to CSV/HTML artifact generated by SysAdminSuite. |

## Guardrails

- Do not store command-console credentials in the repo.
- Do not scrape secured pages unless the client approves the method.
- Do not treat package presence as clinical integration readiness by itself. It is only a software-reference signal.
- Keep reference JSON files immutable by reference id. If the client updates the reference, add a new JSON file instead of silently editing the old one.
