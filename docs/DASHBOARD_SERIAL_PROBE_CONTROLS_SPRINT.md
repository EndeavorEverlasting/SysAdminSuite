# Dashboard Serial Probe Controls Sprint

## Purpose

The backend, runbook, and serial-planning doctrine now support a serial-first Cybernet survey path, but the dashboard controls have not caught up.

This sprint defines the product and implementation contract to bring the dashboard UI in line with the current SysAdminSuite survey workflow.

## Current mismatch

The repo now supports this path:

```text
Alejandro serial list
  -> approved serial-to-host/IP evidence
  -> serial preflight planner
  -> survey/input/serial_preflight/<run_id>/to_probe_targets.txt
  -> PowerShell network preflight
  -> load generated evidence
  -> repeat with local evidence narrowing the next run
```

But the dashboard still primarily presents an older target-file-first flow:

```text
Start Cybernet Survey
  -> Load targets
  -> Network posture
  -> Identity evidence
  -> Reachability
  -> Review results
```

The UI must stop implying that a technician's main choice is manual host/IP targets. The primary survey reality is often:

```text
I have serials. Where did they go?
```

## Product principle

A technician should be able to use the dashboard without asking an AI agent for the next command.

The UI should guide them through:

```text
load Alejandro serial list
  -> load or select approved serial-to-host/IP evidence
  -> generate serial preflight plan
  -> review counts and mystery serials
  -> copy/run staged network preflight command
  -> load output evidence
  -> repeat same planning action until improvement plateaus or review is required
```

## Non-goals

This sprint must not make the dashboard run probes directly.

It must not:

- execute PowerShell commands from the browser
- ping serial strings
- hide network activity
- commit live serials, hostnames, IPs, MACs, or output evidence
- treat reachability as serial identity proof
- treat repeated non-response as proof that a serial does not exist
- replace the workbook/spreadsheet as the source artifact

The dashboard remains an operator guidance, evidence-loading, summary, and command-copy surface.

## Required UI additions

### 1. Serial-first survey entry

Add a clear Cybernet Survey action:

```text
Plan from Alejandro serials
```

This should appear near `Start Cybernet Survey` or inside the Cybernet Survey wizard before the Network posture step.

### 2. Serial planning card

Add a card/panel that explains:

```text
Start with Alejandro's serial list. The suite will not ping serial strings. It will stage only hostnames/IPs that have approved bridge evidence.
```

Required controls:

```text
Serial list file path / selector hint
Evidence bridge file path / selector hint
Ports field with default 135,445,3389,9100
Copy serial preflight command
Copy network preflight command after plan output is loaded
Load serial preflight output
```

### 3. Command generated by UI

The dashboard should be able to show this command shape:

```powershell
Set-Location <SysAdminSuite repo root>
.\survey\sas-serial-preflight-plan.ps1 `
  -SerialFile .\targets\local\alejandro_serials.csv `
  -EvidenceFile .\targets\local\approved_serial_hostname_bridge.csv `
  -Ports 135,445,3389,9100
```

After a serial preflight summary is loaded, it should surface the generated next command from the summary/handoff when available.

### 4. Output files the UI should understand

The dashboard should recognize and summarize:

```text
survey/output/serial_preflight/<run_id>/serial_preflight_plan.csv
survey/output/serial_preflight/<run_id>/review_required.csv
survey/output/serial_preflight/<run_id>/serial_preflight_summary.json
survey/output/serial_preflight/<run_id>/operator_handoff.txt
survey/input/serial_preflight/<run_id>/to_probe_targets.txt
```

The UI should not ask the user to manually inspect all of these first. It should show concise counts and next action.

### 5. Required summary cards

When serial preflight output is loaded, show:

```text
Serials loaded
Probe-ready staged targets
Review-required serials
Mystery serials / no bridge
Next target file path
Next command
Planner network activity: false
```

### 6. Iterative attempt controls

Add a future-ready section for iterative probe handoffs:

```text
Plan next serial probe iteration
```

This can initially be a documentation-backed command card, then later become a parser-backed panel.

It should explain that the command should inspect local evidence, previous attempts, time-of-day diversity, and plateau/no-improvement state before staging the next target file.

## Required dashboard states

Recommended state names:

```text
serial_plan_not_started
serial_plan_command_ready
serial_plan_output_loaded
serial_plan_probe_targets_ready
serial_plan_review_required
serial_plan_mystery_serials_present
serial_plan_no_improvement_plateau
network_preflight_ready
network_preflight_output_loaded
```

## User-facing wording

Use plain operational wording.

Preferred:

```text
Plan from Alejandro serials
Serials are not ping targets.
Only bridged hostnames/IPs are staged for network preflight.
Mystery serials stay in review.
Run the same planning command again after loading new evidence.
```

Avoid:

```text
Fuzzy scan
Discover everything
Ping serials directly
Ignore unresolved serials
Stealth / no traces / bypass monitoring
```

## Parser / evidence-loading requirements

The dashboard should classify serial preflight artifacts distinctly from general CSV evidence.

Recommended file types:

```text
serial-preflight-plan
serial-preflight-review
serial-preflight-summary
serial-preflight-handoff
serial-preflight-targets
serial-iteration-summary
serial-iteration-rollup
```

Minimum detection hints:

```text
serial_preflight_plan.csv
review_required.csv with serial preflight columns
serial_preflight_summary.json
operator_handoff.txt containing SysAdminSuite serial preflight plan
survey/input/serial_preflight/<run_id>/to_probe_targets.txt
```

## Visual review model

Serial planning review should separate rows into buckets:

```text
Ready to probe
Review required / no bridge
Mystery serials
Already confirmed / skip
Repeatedly silent / time-diverse
Possible stale hostname or DNS
Possible wrong subnet or site context
```

Do not bury mystery serials inside a generic warning list. They are often prime targets because they answer the question:

```text
Where did this serial go?
```

## Known risks

### Risk: UI encourages manual target entry

Manual host/IP text boxes are still useful, but they should not be the main path for serial-driven Cybernet work.

Mitigation:

```text
Make serial-first planning the visible Cybernet path. Keep manual target entry under Advanced Tools.
```

### Risk: user thinks serials are pinged directly

Mitigation:

```text
Repeat in UI copy: serials are not ping targets; bridged hostnames/IPs are staged.
```

### Risk: review-required serials disappear

Mitigation:

```text
Show review-required and mystery serial counts every time serial preflight output is loaded.
```

### Risk: repeated failed probes get overinterpreted

Mitigation:

```text
Classify repeated non-response as evidence of non-response, not proof of non-existence.
Require time-diverse attempts before persistent-silence labels.
```

### Risk: dashboard parser mistakes source files for evidence

Mitigation:

```text
Separate source artifacts, planning artifacts, network evidence, and identity evidence in labels and panels.
```

## Suggested implementation order

1. Add dashboard copy/control labels for serial-first planning.
2. Add command card for `sas-serial-preflight-plan.ps1`.
3. Add parser detection for serial preflight summary JSON and plan/review CSVs.
4. Add synthetic samples only.
5. Add dashboard smoke coverage for serial preflight artifacts.
6. Add review buckets for ready-to-probe, review-required, and mystery serials.
7. Add next-command display from summary JSON/operator handoff.
8. Add future placeholder for iterative serial probe handoff.

## Acceptance criteria

Done means:

- The Cybernet Survey UI has a visible serial-first planning path.
- The UI can generate/copy the serial preflight planner command.
- The UI explains that serials are not ping targets.
- The UI accepts and summarizes serial preflight output artifacts.
- Mystery serials and review-required rows are visible as first-class outcomes.
- The UI shows the next command or next handoff after serial preflight output is loaded.
- Dashboard Smoke covers the new sample artifacts and parser paths.
- Pester stays green.
- No live operational evidence is committed.
